# Evaluación Técnica y Estructural: ISO/IEC 27000 (SGSI)

> **Norma:** Familia ISO/IEC 27000 - Sistemas de Gestión de Seguridad de la Información (SGSI).
> **Proyecto:** JM y JS Alimentos LMS v2.0
> **Directorio de Referencia:** `app/Http/Middleware/`, `app/Services/`, `routes/web.php`
> **Fecha de Evaluación:** Junio 2026

La conformidad con la familia de normas ISO/IEC 27000 en el entorno de JM y JS Alimentos se fundamenta en los controles de seguridad aplicados a nivel de código, infraestructura local y gestión de procesos financieros transaccionales.

## 1. Confidencialidad: Controles de Acceso y Datos
- **Autenticación Fuerte:** `AuthController` maneja la autenticación previniendo ataques de fuerza bruta usando el middleware `throttle:5,1` en `/login` (`routes/web.php`), limitando a 5 intentos por minuto.
- **Criptografía de Credenciales:** Todas las contraseñas son codificadas usando Bcrypt (algoritmos Hash de 12 rondas). Laravel prevé nativamente ataques de temporización (*Timing Attacks*) durante la resolución.
- **Control RBAC y ACL:** El acceso al BackOffice (`/admin`) está cerrado por `App\Http\Middleware\AdminMiddleware`. Dentro del ecosistema, los roles (Instructor, Soporte, SuperAdmin) son auditados programáticamente con `middleware('permission:...')`.

## 2. Integridad: Trazabilidad y Consistencia Financiera
- **Protección Transaccional (ACID):** El `PaymentController@process` (`PaymentController.php`) envuelve la creación de los modelos `Sale` y `SaleItem` dentro de `DB::transaction()`. Si una aserción interna se quiebra, la venta completa sufre un *rollback* de base de datos.
- **Sistema de Auditoría Inmutable (`RNF-04`):** El componente estático `AuditService::log()` captura programáticamente las operaciones de los administradores. En el registro se incluyen el User-Agent, Dirección IP (`request()->ip()`), ID de Originador y los cambios realizados pre y post ejecución.
- **Firma Criptográfica Webhook:** Los Webhooks de Stripe recibidos son verificados rigurosamente validando el hash HMAC para garantizar la no repudiación y autenticidad del mensaje financiero.

## 3. Disponibilidad: Resiliencia del Sistema
- **Defensas Anti-Lockout (`RNF-06`):** Una rutina programática implementada en `Admin\UserController@toggleAdmin` prohíbe que el último administrador activo sea degradado, lo cual inhabilitaría eternamente al panel de administración.
- **Idempotencia Transaccional:** El Webhook evalúa si el pago ya fue ejecutado comprobando la clave primaria transaccional (ej. ID de Checkout de Stripe). Si fue ejecutado, la petición se ignora para prevenir sobrecarga y facturación múltiple ante latencia de red.

## 4. Gestión de Secretos y Defensas Web (OWASP)
- **Configuración Sensible:** No existen secretos en el código duro. Credenciales críticas como `STRIPE_SECRET` y `GEMINI_API_KEY` son leídas del environment (`.env`).
- **Sanitización Automática XSS y SQLi:** El motor Blade procesa sentencias `{{ }}` pasando funciones `htmlspecialchars()` de PHP. Todos los accesos de base de datos se hacen a través del ORM Eloquent, que utiliza bindings por defecto, inutilizando la inyección SQL clásica.
