# Evaluación Técnica y Estructural: ISO/IEC 25000 (SQuaRE)

> **Norma:** ISO/IEC 25000 - Requisitos y Evaluación de la Calidad del Producto de Software (SQuaRE).
> **Proyecto:** JM y JS Alimentos LMS v2.0
> **Directorio de Referencia:** `app/`, `tests/`, `routes/`
> **Fecha de Evaluación:** Junio 2026

La norma ISO/IEC 25000 establece un marco riguroso para la evaluación de la calidad del software. Este documento detalla la justificación técnica del cumplimiento del proyecto LMS v2.0 basándose en la evidencia del código fuente.

## 1. Adecuación Funcional (Functional Suitability)
- **Completitud y Corrección Funcional:** La lógica de negocio está completamente respaldada por una suite de **89 pruebas automatizadas** con **426 aserciones**. Clases como `Tests\Feature\PaymentStripeTest` y `Tests\Feature\PublicCourseCatalogTest` validan desde la visualización de cursos hasta la integración del webhook de Stripe (`POST /stripe/webhook`), garantizando la creación de instancias de `Enrollment`.
- **Pertinencia Funcional:** La implementación del `App\Http\Controllers\Api\ChatController` con Google Gemini 2.5 Flash cubre específicamente el requisito de soporte asíncrono, operando como un widget en React.

## 2. Eficiencia de Desempeño (Performance Efficiency)
- **Comportamiento Temporal:** Eloquent ORM utiliza "Eager Loading" explícito (ej. `Course::with(['category'])`) para mitigar consultas N+1. La compilación de activos estáticos es orquestada por **Vite 7** y **Tailwind CSS v4** mediante HMR, garantizando la optimización de código entregado al cliente (React 19).
- **Utilización de Recursos:** La persistencia temporal en `session()->get('cart')` previene múltiples lecturas de disco innecesarias durante las sesiones del cliente en el `CartController`.

## 3. Compatibilidad e Interoperabilidad (Compatibility)
- **Interoperabilidad de Pasarelas y APIs:** El componente `App\Services\StripeService` gestiona las sesiones de `Stripe\Checkout\Session` interactuando de manera transparente con la API externa, devolviendo los webhooks al método `PaymentController@webhook` en total aislamiento asíncrono.

## 4. Usabilidad (Usability)
- **Retroalimentación Constante:** Los controladores web retornan respuestas HTTP 302 con *Flash Sessions* (`redirect()->with('status')`) renderizadas mediante Blade Toasts y alertas responsivas.
- **Protección contra Errores de Usuario:** Validaciones tempranas mediante FormRequests y `validate()` detienen transacciones malformadas antes de impactar los procesos pesados de base de datos.

## 5. Fiabilidad y Recuperabilidad (Reliability)
- **Tolerancia a Fallos y Transacciones ACID:** El método `process()` en `PaymentController` envuelve la creación de entidades financieras dentro de un bloque **`DB::transaction()`**. Si un `SaleItem` falla, se realiza un rollback automático (`RNF-05`).
- **Resiliencia Externa:** El `ChatController` implementa bloques `try-catch` para capturar latencia o errores 500 de la API de Gemini, retornando un `fallback` amigable sin romper el frontend.

## 6. Seguridad (Security)
- **Criptografía y Secretos:** Las contraseñas utilizan `Hash::make()` (Bcrypt). Las claves de IA y Stripe están inyectadas mediante el archivo nativo `.env` y registradas en `config/services.php`.
- **Control de Acceso y Trazabilidad:** Implementación estricta de `AdminMiddleware` y `AuditService::log()` para asentar registros inmutables en base de datos.

## 7. Mantenibilidad (Maintainability)
- **Modularidad:** Adopción del patrón MVC y Clean Architecture. Las rutas administrativas están fuertemente aisladas de los endpoints públicos. El uso de `Laravel Pint` preserva el estándar PSR-12.

## 8. Portabilidad (Portability)
- **Agnosticismo del Entorno:** La sustitución limpia del driver de sesión y de base de datos a `sqlite: memory` (en el archivo `phpunit.xml`) atestigua la alta portabilidad del monolito hacia contenedores Docker (Nginx, PHP-FPM) en producción.
